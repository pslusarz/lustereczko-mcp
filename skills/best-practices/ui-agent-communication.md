# UI ↔ Agent Communication

## Overview

MCP Apps protocol is limited to `sendMessage` and `updateModelContext`, which provide a very limited experience. This document describes the richer communication techniques available and when to use each.

General rule: pick the simplest protocol that covers all your app's use cases and use it consistently throughout. Don't mix protocols. Exception: logging can be mixed with any protocol.

---

## Channel IDs — the isolation primitive

Every `display_ui_to_user` call creates a new UI render. Because the host may run multiple server processes, and because replaced HTML does not reliably kill old JavaScript contexts, each render needs its own isolated communication channel.

**The agent assigns the channel ID** at the moment it generates the HTML for `display_ui_to_user`. It embeds the ID as a literal constant in the `<script>` block. Both the agent and the UI therefore know the channel ID from the start — no handshake is required.

```python
# Agent side — before calling display_ui_to_user
import time
channel_id = f"ch-{int(time.time() * 1000)}"   # e.g. "ch-1749312345678"

# Embed it in the HTML fragment
html = f"""
<script>
  const MY_CHANNEL = {json.dumps(channel_id)};
  // ... rest of UI code using MY_CHANNEL
</script>
"""
display_ui_to_user(html_fragment=html)
# Now the agent can immediately call notify_ui(channel_id=channel_id, ...)
# without waiting for any ready event.
```

### Channel ID rules

- Generated by the agent, not the UI.
- Format: alphanumeric, hyphens, underscores, 1–64 characters (e.g. `ch-1749312345678`).
- Scoped to one `display_ui_to_user` call. A new call gets a new ID.
- The agent tracks the current channel ID and uses it for all `notify_ui` and `poll_agent_messages` calls until it issues the next `display_ui_to_user`.

### Multi-panel operation

If the host renders multiple panels simultaneously (or a new render doesn't destroy the old one), each panel operates on its own channel. The agent can hold multiple channel IDs and address each panel independently. Old panels that are still alive simply keep polling their own channel — they neither interfere with nor receive traffic for the new channel.

```python
# Two independent panels
panel_a = f"ch-{int(time.time() * 1000)}-a"
panel_b = f"ch-{int(time.time() * 1000)}-b"

# Agent can send to each independently
notify_ui(channel_id=panel_a, event="update", data={"section": "header"})
notify_ui(channel_id=panel_b, event="update", data={"section": "sidebar"})
```

### The `ready` signal (optional)

The UI may send a `ready` event on startup to signal that it has connected and begun polling. This is purely informational — the agent already knows the channel ID and can start sending immediately. It is useful if the agent wants to confirm the UI is alive before sending a burst of messages.

```js
// UI — optional, send after first successful poll
window.app.callServerTool({
  name: 'notify_agent',
  arguments: { event: 'ready', channel_id: MY_CHANNEL, data: { channel_id: MY_CHANNEL } }
});
```

## MCP App primitives

### updateModelContext

`window.app.updateModelContext()` silently inserts context into the chat, which is attached when the user sends their next prompt. Only useful for the simplest interactions where no immediate agent response is needed.

### sendMessage

`window.app.sendMessage()` populates the user's input box with a message. The user sees it and decides when (or whether) to submit. Useful when the UI collects structured choices and needs to surface them clearly to the user before the agent acts. Confusing for large payloads. Do not mix with the polling protocol.

### oncalltool — do not use

`oncalltool`/`onlisttools` require the Claude desktop host to relay the agent's `tools/call` request from the agent's MCP connection to the UI's SSE connection. This relay is not implemented in Claude desktop 1.0.0. Setting `oncalltool` triggers a capability negotiation that fails with "Client does not support tool capability", and may also break `callServerTool` for the remainder of the session.

**Do not attempt to implement this mechanism.** Use `notify_ui` + `poll_ui_messages` for agent→UI signaling instead.

## notify_ui — agent→UI signaling

`notify_ui` enqueues an event on the channel's file-backed queue. The UI drains it by polling `poll_ui_messages` on a timer. Each channel has its own queue file; messages for one channel are never visible to another.

**Agent sends:**

```python
notify_ui(
    event="ping",
    channel_id=channel_id,
    data={"id": 1, "text": "Hello from the agent"},
)
```

**`poll_ui_messages` returns** (as `content[0].text`, JSON-encoded):

```json
[
  {"event": "ping", "data": {"id": 1, "text": "Hello from the agent"}},
  {"event": "ping", "data": {"id": 2, "text": "Second message"}}
]
```

Returns all pending items in FIFO order and atomically clears the channel's queue. Returns `[]` when empty.

**Minimal UI polling loop:**

```js
const MY_CHANNEL = "ch-1749312345678";   // injected by agent at render time

function normalize(d) {
  // data may arrive as a pre-serialized string on some FastMCP versions
  if (d == null || typeof d !== 'string') return d;
  try { return JSON.parse(d); } catch(e) { return d; }
}

function poll() {
  window.app.callServerTool({
    name: 'poll_ui_messages',
    arguments: { channel_id: MY_CHANNEL }
  })
    .then(result => {
      const text = result?.content?.[0]?.text ?? result;
      const items = JSON.parse(typeof text === 'string' ? text : JSON.stringify(text));
      items.forEach(evt => handleEvent(evt.event, normalize(evt.data)));
    })
    .catch(err => console.warn('poll error', err))
    .finally(() => setTimeout(poll, 1500));
}

setTimeout(poll, 500);   // short delay for app.connect() to settle
```

**Notes:**
- `callServerTool` takes `{ name, arguments }` — not positional args.
- Always use `.finally(() => setTimeout(poll, N))` so polling continues after errors.
- 1500 ms is a safe interval; the network call itself adds latency on top.
- `data` values may deserialize as strings on some FastMCP versions — always normalize.
- Because MY_CHANNEL is a literal embedded at render time, old UI instances polling a stale channel ID naturally receive nothing — no kill-switch logic needed.

Full App handler reference: [apps.extensions.modelcontextprotocol.io/api/classes/app.App.html](https://apps.extensions.modelcontextprotocol.io/api/classes/app.App.html)

## notify_agent — UI→agent signaling

The symmetric counterpart to `notify_ui`. The UI calls `notify_agent` via `callServerTool`; the agent drains with `poll_agent_messages`. Each channel has its own agent queue file.

**UI posts an event:**

```js
await window.app.callServerTool({
  name: 'notify_agent',
  arguments: {
    event: 'ack',
    channel_id: MY_CHANNEL,
    data: { id: 5 }
  }
});
```

**Agent drains the queue:**

```python
items = poll_agent_messages(channel_id=channel_id)
# items: [{"event": "ack", "data": {"id": 5}}, ...]
```

The agent should poll after any significant UI interaction window, or run a scheduled check. Only messages for the specified `channel_id` are returned.

## Complete protocol flow

```
Agent                                   UI
──────                                  ──
channel_id = "ch-1749312345678"
display_ui_to_user(html with            ← renders, MY_CHANNEL = "ch-1749312345678"
  MY_CHANNEL embedded)                    starts polling poll_ui_messages(channel_id)

                                        → notify_agent(event="ready",        [optional]
                                            channel_id=MY_CHANNEL)

notify_ui(event="ping",              →
  channel_id=channel_id,
  data={"id": 1})

                                        handles event, auto-acks:
                                        → notify_agent(event="ack",
                                            channel_id=MY_CHANNEL,
                                            data={"id": 1})

poll_agent_messages(channel_id)      ←
# [{"event": "ack", "data": {"id": 1}}]
```

## Adding custom backend tools

Unless the data is very simple, keep data fetching out of the UI HTML. Register a custom tool via `add_custom_tool`; the UI calls it via `callServerTool("run_custom_tool", ...)`.

```python
def run():
    return open("/absolute/path/to/file.txt").read()
```

Use absolute paths — the server's working directory is not predictable.

## The log back-channel

Instrument the app with debug output. The UI calls `write_server_log` to emit state information; the agent calls `tail_server_log` to read it. Host capabilities negotiated at app startup are already written to the log automatically.

---

## Why channel IDs solve the multi-process problem

The host may start multiple server processes. Without channel isolation, all processes share a single queue file, so a `notify_ui` written by one process could be consumed by a different process's `poll_ui_messages`, and UI acks could be written to a process whose agent is never read. Channel IDs solve this by giving each communication session its own file pair (`channels/ui_{channel_id}.json`, `channels/agent_{channel_id}.json`). The file-level `flock` on each file ensures atomic reads and writes under concurrent access from any number of processes.
